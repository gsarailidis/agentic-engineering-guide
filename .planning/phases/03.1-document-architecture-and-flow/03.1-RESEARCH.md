## Summary

This phase is not a content-expansion phase. It is a document-spine phase over [AgenticEngineeringGuide.md](/home/nick/wd/agentic_guide/AgenticEngineeringGuide.md): make the guide read as one argument, clarify each section's job, and create a clean structural slot for Phase `03.2` before the final GSD section. The authoritative sources are the phase context, roadmap, requirements, and current guide draft; no external library research is needed because the work is editorial and repo-specific.

The current draft already contains the right major ingredients, but the argument is split across too many layers. The introduction explains scope but does not yet distinguish development-time use from production-system use. Sections `I-III` are generic and partly disconnected from the later environment-first body. Sections `IV-VI` are stronger, but the guide currently repeats loop/architecture ideas across multiple places and gives the engineering-workflow perspective less structural weight than the new phase requires.

The right planning posture is: treat the guide as a controlled refactor of section roles, not a rewrite. Start by defining the target spine and section contracts, then perform only the minimum intro and `I-III` edits needed to support that spine, leave substantive `03.2` content out, preserve GSD as the final worked framework example, and keep Phase `4` focused on tightening rather than reopening architecture.

**Primary recommendation:** Plan `03.1` as a sequence of section-role mapping, minimal pre-`IV` restructuring, and transition rewriting, with an explicit hold-the-line boundary that reserves a top-level slot for Phase `03.2` without drafting it yet.

### Pattern 1: Introduction As Contract And Reading Map
**What:** Use the introduction to define subject, scope, exclusions, what the guide covers, and the development-time versus production-system distinction.
**When to use:** At the top of the guide and nowhere else as a repeated contrast.
**Example:**
```markdown
## Introduction

This guide is about terminal-native local agents used for engineering work.

It focuses on development-time environments where the model can inspect a workspace,
run commands, use tools, and leave durable artifacts. That is different from a
production agent system embedded inside a shipped product.

What follows:
- the basic concepts and vocabulary
- the environment and operating patterns
- one concrete harness example
- one workflow-framework example
```

### Pattern 2: Section Roles Must Form A One-Way Ladder
**What:** Each major section should do a distinct job and hand off once to the next layer: orientation -> concepts -> environment patterns -> concrete harness -> workflow framework.
**When to use:** When reworking `I-III`, `IV`, `V`, and `VI` so they stop competing with each other.
**Example:**
```text
Introduction  -> what this guide is, what it is not, how to read it
I-III         -> minimal conceptual prerequisites and vocabulary
IV            -> environment-level operating patterns
V             -> Claude Code as the concrete harness example
03.2 slot     -> framing/research/spec-driven engineering section
VI            -> GSD as the final workflow-framework example
```

### Pattern 3: Reserve Phase 03.2 With A Structural Bridge, Not Substantive Content
**What:** Add only enough transition language to make the next top-level section feel expected.
**When to use:** At the end of the Claude Code section or wherever the final spine places the `03.2` bridge.
**Example:**
```markdown
The next layer is not another tool surface but the workflow design built on top of
these environments: framing the task, gathering evidence, shaping the plan, and
turning that work into explicit specs. That belongs before any concrete workflow
framework example.
```

### Anti-Patterns to Avoid
- **Cross-cutting the dev-time vs production distinction:** State it once in the introduction and stop repeating it through unrelated sections.
- **Duplicated loop explanations:** `II`, `IV`, `V`, and `VI` currently all touch loops; each section needs a narrower role.
- **Rewriting `03.2` early:** This phase should make room for the section, not draft its substantive body.
- **Letting GSD become the architectural center:** GSD stays last and illustrative, not the guide's organizing principle.
- **Treating Phase 4 as a cleanup bucket:** Structural problems should be solved here so Phase `4` can stay an editorial tightening pass.

## Common Pitfalls

### Pitfall 1: Unresolved authority to edit before `# IV`
**What goes wrong:** The plan assumes intro and `I-III` edits are allowed, but the repository still contains a hard preservation rule and a snapshot verifier for the pre-`IV` prefix.
**Why it happens:** Phase `03.1` was added later and expands the editable area without explicitly replacing the earlier guardrail.
**How to avoid:** Make the first planning step an explicit decision: either treat the new phase context as authorization to revise intro / `I-III`, or add a mini-confirmation task before implementation.
**Warning signs:** The plan includes pre-`IV` edits but still calls the existing verifier as if no changes are permitted.

### Pitfall 2: Leaving section roles blurry
**What goes wrong:** `I-III` stay generic, `IV` repeats engineering ideas, `V` repeats the loop, and `VI` repeats workflow-control concepts without a clean escalation.
**Why it happens:** The draft accumulated by phase rather than by final document role.
**How to avoid:** Define a one-sentence contract for every top-level section before editing prose.
**Warning signs:** Multiple sections answer the same question, especially around "what is the loop?" or "what makes these systems different from chat?"

### Pitfall 3: Smuggling Phase 03.2 into Phase 03.1
**What goes wrong:** The architecture pass starts writing full material on framing, research, specs, and workflow design.
**Why it happens:** Those topics naturally follow the new engineering-workflow emphasis.
**How to avoid:** Add only the structural slot and bridge language needed to support the later section.
**Warning signs:** New subsections appear before GSD that materially explain research/spec-driven engineering rather than just setting up the next phase.

### Pitfall 4: Over-correcting into a full rewrite
**What goes wrong:** The phase rewrites the whole document, making later verification and scope control harder.
**Why it happens:** Architecture problems can tempt a "start over cleanly" response.
**How to avoid:** Work from a target spine and section-role diff, not from a blank page.
**Warning signs:** Planning tasks are framed as "rewrite the guide" rather than "reassign section roles and transitions."

### Pitfall 5: Weak transition into GSD
**What goes wrong:** Even after restructuring, GSD still feels appended rather than earned.
**Why it happens:** The guide moves from environment patterns straight into framework detail without making workflow design equally central first.
**How to avoid:** Ensure the spine creates a clear "environment -> harness -> workflow design -> workflow framework" progression.
**Warning signs:** GSD still appears as a sudden tooling appendix instead of the final concrete layer.
